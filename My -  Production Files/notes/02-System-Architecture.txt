========================================================================
02 - DETAILED SYSTEM ARCHITECTURE & WORKFLOW
========================================================================

1. ARCHITECTURAL OVERVIEW
This prototype operates as a fully decoupled frontend application. Instead of hardcoding data or bypassing security via local global variables (e.g., `window.SCHOOL_DATA`), the architecture enforces a strict client-server communication pattern using a local HTTP Server, a Mock API Gateway, and a JSON-based database structure.

2. COMPONENT BREAKDOWN

A. The Data Layer (The Database)
- Location: `/database/records/*.json`
- Function: Acts as the persistent storage layer. Each JSON file represents a single document record (a university). The shape of this data is strictly enforced by `/database/database_schema.json`, which was derived directly from the `DreamCollege_CDS_Field_Catalog Final V1.0.xlsx` specification.

B. The Configuration Layer
- Location: `/database/config.json`
- Function: Stores environmental variables and active API Keys. It explicitly lists `DREAM_COLLEGE_API_v1_847294` as an active key. This file acts as the source of truth for the API Gateway's authorization checks.

C. The API Gateway (The Middleware)
- Location: `/database/api_client.js`
- Function: Exports the `DreamCollegeAPI` class. It acts as the middleware proxy between the Presentation Layer and the Data Layer. It intercepts data requests, validates the provided API key against `config.json`, and handles the asynchronous file fetching and error reporting (401 Unauthorized, 404 Not Found).

D. The Presentation Layer (The UI)
- Location: `College-List-Demo.html`
- Function: The HTML/CSS/JS frontend. It is completely agnostic to the file system. It instantiates the `DreamCollegeAPI`, requests data via `dbAPI.getSchool(schoolId)`, and uses a declarative DOM-manipulation function (`renderSchoolData`) to update the DOM elements based on the returned schema.

3. THE EXECUTION FLOW
Step 1: The user runs `Run-Project.bat`, which spawns a Python HTTP server on port 8000 and opens the browser. (This bypasses the CORS `file://` protocol restrictions).
Step 2: The browser parses `College-List-Demo.html`.
Step 3: At the bottom of the HTML document, the `api_client.js` script is executed.
Step 4: The presentation layer invokes `const dbAPI = new DreamCollegeAPI(...)`.
Step 5: The `DreamCollegeAPI` constructor immediately triggers an asynchronous authorization check, fetching `config.json` to validate the key.
Step 6: The `DOMContentLoaded` event fires in the HTML, triggering `fetchSchoolData()`.
Step 7: `fetchSchoolData` calls `await dbAPI.getSchool(fileName)`.
Step 8: The API client waits for the authorization promise to resolve. If authorized, it fetches the raw JSON from `/database/records/` and returns the parsed object.
Step 9: The HTML executes `renderSchoolData(data)`, updating all visual charts, GPA graphs, and text fields to reflect the selected university.

========================================================================
03 - DETAILED DATABASE, API KEYS, AND SCHEMA DOCUMENTATION
========================================================================

1. THE DATABASE STRUCTURE
The application stores its data in the `database/records/` directory. Each file within this directory corresponds to a single university (e.g., `HarvardUniversity_CDS_2024-2025.json`). 
Unlike standard implementations where the frontend pulls directly from unstructured files, this directory operates as a mocked NoSQL document database. The frontend is forbidden from traversing or querying this folder directly.

2. THE CONFIGURATION AND API KEYS
To simulate enterprise-level security, we introduced a centralized configuration file located at `database/config.json`.
- Purpose: Stores the valid API keys and environmental settings.
- Structure:
  {
    "api_keys": {
      "DREAM_COLLEGE_API_v1_847294": {
        "owner": "Julia Laing",
        "access_level": "admin",
        "status": "active"
      }
    }
  }
- Security Flow: The API client (`api_client.js`) reads this config file during initialization. It compares the API key provided by the HTML file against the keys in this JSON file. If the key is not present or not marked "active", the database connection is rejected.

3. THE SCHEMA ENFORCEMENT (database_schema.json)
The original AI-generated schema provided was replaced with a highly strict, industrial-grade schema (`database/database_schema.json`). This schema was built directly from the specifications in `DreamCollege_CDS_Field_Catalog Final V1.0.xlsx`.

Schema Architecture Breakdown:
- `cds_meta`: Contains primary identifiers (school_name, ipeds_id, cds_year). The IPEDS ID acts as the primary key for relational mapping.
- `cds_admissions`: Contains nested objects for `class_size` (applicant/admitted counts, calculated yield rates), `early_programs`, `test_scores` (25th/75th percentiles), `gpa_distribution`, and the 18 holistic `factors` (rigor, essay, alumni relation, etc.).
- `cds_academics`: Contains institutional metrics (faculty ratio, undergrad enrollment, setting).
- `cds_financials`: Contains segmented cost data (in-state vs out-of-state tuition, room and board) and aid data (average need-based packages).
- `cds_outcomes`: Contains performance metrics (retention rate, 4-year and 6-year graduation rates).

The DOM elements in `College-List-Demo.html` expect exactly this structural hierarchy. For instance, the UI accesses the holistic factors via `data.cds_admissions.factors.rigor_of_secondary_record`. Any deviation from this schema will cause immediate rendering failures in the UI.
